# 11. 디자인 플레이북

이 플레이북은 Taebin Kim의 포트폴리오를 위해 작성되었으나,
향후 모든 프로젝트에서 독립적으로 재사용할 수 있도록 보편적 원칙 중심으로 서술한다.

---

## 디자인 철학

### 핵심 원칙

구조가 인상이다.
꾸밈보다 계층이 먼저다.
정보의 위계가 시각적 위계와 일치해야 한다.
여백은 장식이 아니라 구조다.
색은 의미를 전달할 때만 사용한다.

### 설계 순서

1. 무엇을 전달하려는가를 먼저 결정한다
2. 그것을 가장 단순한 형태로 표현한다
3. 그 다음 시각적 계층을 더한다
4. 마지막으로 인터랙션을 더한다

---

## 브랜드 무드

### 이 포트폴리오의 무드

기본: Refined Dark Exhibition (정제된 다크 전시)
보조: Minimal Light Gallery (미니멀 라이트 갤러리)

무드 키워드:
- 정제된
- 영화적이되 절제된
- 기술적이되 따뜻한
- 몰입적이되 읽기 쉬운

무드 안티패턴:
- 게임적
- 과잉 애니메이션
- 컬러풀한 클러터
- 장식을 위한 장식

---

## Black/White 기반 테마 전략

### 원칙

1. 검정과 흰색이 기반이다. 회색은 계층을 만든다.
2. 액센트 색상은 한 가지 역할만 한다. 주의를 끌기 위해서다.
3. 다크 테마에서 흰색은 강조이고, 라이트 테마에서 검정이 강조이다.
4. 색조(hue)가 있는 색상은 피드백 상태(성공, 오류, 경고)에만 사용한다.

### 테마 전환의 목표

다크 → 라이트 전환은 단순한 색상 반전이 아니다.
분위기의 전환이다. 어두운 전시관 → 밝은 갤러리.
두 모드 모두 동일한 품질과 계층을 유지해야 한다.

## 다크/라이트 시맨틱 컬러 토큰

### 토큰 사용 규칙

1. 코드에서 CSS 변수나 Tailwind 토큰 이름만 사용한다
2. 원시 색상값(#ffffff, rgb(0,0,0))을 직접 코드에 쓰지 않는다
3. 새로운 색상이 필요할 때는 토큰을 먼저 정의하고 사용한다
4. 토큰 이름은 항상 목적(role)을 설명한다, 색상값이 아니라

### 표준 토큰 목록

surface 토큰:
- surface-base: 페이지 기본 배경
- surface-elevated: 카드, 패널, 드롭다운 배경
- surface-overlay: 모달, overlay 배경 (반투명)
- surface-input: 입력 필드 배경

text 토큰:
- text-primary: 주요 본문 텍스트
- text-secondary: 보조 텍스트, 캡션, 레이블
- text-disabled: 비활성화 상태 텍스트
- text-inverse: 어두운 배경 위의 밝은 텍스트 (또는 반대)
- text-link: 링크 텍스트

border 토큰:
- border-default: 일반 구분선, 카드 테두리
- border-strong: 강조 구분선, 포커스된 카드
- border-focus: 키보드 포커스 링

accent 토큰:
- accent-primary: 주요 CTA, 선택 상태, 링크 hover
- accent-secondary: 보조 CTA, 약한 강조
- accent-muted: 배지, 태그 배경, 매우 약한 강조

feedback 토큰:
- feedback-success
- feedback-warning
- feedback-error
- feedback-info

## 타이포그래피

### 폰트 선택 기준

1. 시스템 폰트 또는 가변 폰트(variable font)를 우선 고려한다
2. 디스플레이와 본문에 같은 폰트를 사용하되 웨이트로 구분한다
3. 모노스페이스는 코드, 기술 태그에만 제한적으로 사용한다
4. 한국어가 포함되면 한국어 폰트의 로딩 전략을 별도로 설계한다

### 타입 스케일 원칙

스케일은 균등 증가가 아닌 역할 기반으로 결정한다.
- 히어로 이름: 시선을 잡는 단 하나의 크기
- 섹션 제목: 계층 구분
- 본문: 읽기 편한 크기
- 레이블/태그: 작지만 읽을 수 있는 크기

### 행간 규칙

- 대형 디스플레이(히어로): 1.0 ~ 1.1
- 제목: 1.2 ~ 1.3
- 본문: 1.6 ~ 1.7
- UI 요소: 1.2 ~ 1.4

## 레이아웃 시스템

### 컨테이너

```
max-width: 1280px (넓은 레이아웃)
max-width: 960px (본문 레이아웃)
max-width: 720px (좁은 레이아웃, 상세 페이지)
padding: 24px (모바일), 48px (태블릿), 80px (데스크톱)
```

### 그리드

프로젝트 아카이브: 3열 (데스크톱) → 2열 (태블릿) → 1열 (모바일)
기술 스택 태그: flex-wrap
overlay: 화면 오른쪽 40% 또는 중앙

### 간격 단위

기본 단위: 4px (0.25rem)
표준 간격: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128px

## 컴포넌트 원칙

### 버튼

1. 한 화면에 Primary 버튼은 하나가 원칙이다
2. 버튼 텍스트는 동사로 시작한다 (보기, 열기, 보내기)
3. 아이콘만 있는 버튼은 반드시 aria-label을 가진다
4. 로딩 상태를 시각적으로 명확히 표현한다

### 카드

1. 카드는 한 가지 주요 정보를 전달한다
2. hover 상태는 미세하게 표현한다 (이동 2-4px, border 강화)
3. 카드 내부에서 폰트 크기를 작게 하지 않는다

### 입력 필드

1. placeholder 텍스트는 레이블을 대체하지 않는다
2. 에러 메시지는 필드 바로 아래에 표시한다
3. 포커스 상태를 border-focus 토큰으로 명확히 표시한다

## 3D 시각 규칙

### 허용

- 오브젝트의 느린 자전과 부유 모션
- hover 시 미세한 스케일 업 (1.0 → 1.1)
- focus 시 카메라의 부드러운 이동
- 테마에 따른 material과 조명 조정

### 금지

- 빠르고 화려한 애니메이션
- 파티클이 화면을 채우는 과잉 이펙트
- 3D 씬이 HTML 가독성을 방해하는 경우
- 오브젝트가 서로 겹치거나 너무 밀집된 배치

### 오브젝트 패밀리 규칙

각 오브젝트는 카테고리 언어를 따라야 한다.
카테고리 언어 밖의 자의적 형태를 만들지 않는다.
오브젝트 간 시각 언어는 통일성을 가지되, 패밀리 특성으로 구분된다.

---

## 상호작용 규칙

### 애니메이션 지속 시간

- 즉각 피드백 (버튼 hover 등): 100-150ms
- UI 트랜지션 (overlay 열기): 200-300ms
- 카메라 이동: 800-1200ms
- 인트로 시퀀스: 2000-3000ms

### 이징 (Easing)

- 요소 등장: easeOut
- 요소 사라짐: easeIn
- 카메라 이동: easeInOut (cubic)
- hover 상태: linear 또는 easeOut (빠르게)

### 감소된 모션 규칙

prefers-reduced-motion: reduce 감지 시:
- 3D 부유 애니메이션 제거
- 페이지 트랜지션 즉시 처리
- overlay 열기/닫기 즉시 처리
- 카메라 이동 즉시 처리 또는 매우 짧게

## 반응형 원칙

### 브레이크포인트

- mobile: 0 ~ 767px
- tablet: 768px ~ 1023px
- desktop: 1024px ~

### 모바일 우선 vs 데스크톱 우선

이 포트폴리오는 데스크톱 경험이 기본이다.
그러나 모바일에서도 완전한 콘텐츠 경험을 제공해야 한다.
모바일에서 3D를 제거하더라도 콘텐츠 접근성은 유지한다.

### 반응형 텍스트

디스플레이 폰트는 뷰포트에 따라 조정된다.
```
히어로 이름:
  mobile: 2.5rem
  tablet: 3.5rem
  desktop: 4.5rem
```

## Do / Don't

### Do

- 계층을 명확히 한다
- 여백을 구조적으로 사용한다
- 토큰을 통해 색상을 관리한다
- 인터랙션에 목적을 부여한다
- 두 테마 모두에서 일관된 경험을 보장한다
- 플레이스홀더를 빈 공간이 아닌 명확한 상태로 표현한다
- 실명(Taebin Kim)을 중심에 두고 브랜딩한다

### Don't

- 색으로 장식하지 않는다
- 애니메이션을 과도하게 사용하지 않는다
- 한 화면에 너무 많은 정보를 넣지 않는다
- 모바일에서 3D를 강제 로딩하지 않는다
- 하드코딩된 색상값을 코드에 직접 쓰지 않는다
- 테마 전환 시 flash가 발생하도록 두지 않는다
- 플레이스홀더를 빈 상태(empty state)로 방치하지 않는다
