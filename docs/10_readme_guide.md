# 10. README 작성 가이드

## README 목적

이 포트폴리오의 README는 두 가지 목적을 동시에 달성해야 한다.

1. 기술 문서: 시스템 구조, 실행 방법, 기술 스택을 명확히 설명한다
2. 포트폴리오 표현: Taebin Kim의 설계 방식과 사고 과정을 보여준다

README 자체가 포트폴리오의 일부이다.
코드를 보기 전에 README를 읽는 사람도 이 프로젝트의 가치를 이해할 수 있어야 한다.

---

## README 작성 규칙

### 언어

한국어로 작성한다.
영문 기술 용어(Next.js, Docker, PostgreSQL 등)는 원어 표기를 유지한다.

### 형식

- 마크다운 헤딩 계층을 명확하게 사용한다 (H1, H2, H3)
- 코드 블록은 반드시 언어 지정을 포함한다
- 긴 설명은 표 형식으로 정리한다
- 이미지 삽입 위치를 미리 지정한다 (placeholder 방식)
- 장식적 마크다운 (* 강조, 이모지 등)은 최소화한다
- 각 섹션은 "왜"로 시작하고 "무엇"과 "어떻게"로 이어진다

---

## 포트폴리오용 설명 방향

README의 각 섹션에서 다음 질문에 답한다.

이 프로젝트를 왜 만들었는가:
- 단순한 포트폴리오가 아닌 이유
- 3D 인터랙션을 선택한 이유
- 풀스택 구조를 선택한 이유

어떻게 설계했는가:
- 구현 전 문서 정의
- 데이터 구조와 사용자 경험을 함께 설계
- 3D와 HTML 레이어의 책임 분리

무엇을 배웠는가:
- 기술적 도전과 해결 과정
- 설계 결정의 트레이드오프
- 향후 개선 방향

## 시스템 구조 서술법

시스템 구조를 설명할 때는 다음 순서를 따른다.

1. 전체 구성 요소 나열 (한 문장씩)
2. 계층 간 데이터 흐름 설명 (사용자 요청 → 응답 흐름)
3. 중요 설계 결정과 그 이유 설명
4. 코드 구조와 폴더 설명

기술 명세를 나열하기 전에 "왜 이 구조를 선택했는가"를 먼저 설명한다.

## 트러블슈팅 작성 방향

트러블슈팅 섹션은 다음 형식을 따른다.

```
### 문제: [문제 제목]

증상: 어떤 상황에서 발생하는지
원인: 왜 발생하는지
해결: 어떻게 해결하는지 (코드 예시 포함)
```

포함할 트러블슈팅 항목:

3D 씬 관련:
- WebGL 미지원 환경 처리
- 3D 씬과 HTML overlay 동기화 문제
- 모바일에서 성능 이슈

배포 관련:
- Docker 네트워크 연결 문제
- 마이그레이션 실패 처리
- 환경변수 누락 시 동작

테마 관련:
- 서버/클라이언트 테마 flash 문제
- 3D 씬 테마 전환 동기화
